# 사람을 위한 코드 작성법

> 프로그래밍은 `소통 활동`이다. <br>
이는 컴파일러와의 소통만을 의미하는게 아니다. <br>
컴파일러는 어떤 코드든 이해할 수 있지만 그렇다고 해서 작성자의 의도를 파악할 수 있는것은 아니다. <br>
소프트웨어 엔지니어는 코드를 읽는 사람이 이해할 수 있는 코드를 작성해야하며, 명확한 의도를 전달할 수 있는 코드를 작성해야 한다.

<br>

## 좋은 코드란?

- 좋은 코드의 기준은 굉장히 주관적이지만, 내가 생각하는 좋은 코드는 마치 책을 읽는 것처럼 자연스럽게 읽히는 코드다.
- 문단이 적절히 나뉘어 있고, 띄어쓰기가 잘 맞춰져 있으며, 한 호흡에 읽을 수 있을 만큼 길지 않아야 한다.
- 또한 같은 추상화 레벨로 코드가 구성되어야 한다. 예를들어 하나의 유즈케이스를 처리한다면 `입력값 유효성 검증 → 외부 데이터 호출 → 비즈니스 로직 수행 → 사후 처리`와 같이 동일한 레벨의 흐름으로 읽혀야 한다.
- 각 단계의 세부 구현도 중요하지만, 전체 맥락의 흐름이 더 우선되어야 한다.

<br>

#### 기존 프로젝트에 지표 추가하기

- 기존 프로젝트에 지표(린트, 테스트 커버리지 등)를 추가하면 감당할 수 없을 정도로 많은 경고가 발생하여 팀원들의 사기가 꺾이는 역효과가 발생할 수 있으니 너무 많은 것을 한번에 수정하려고 하기 보다는 점진적으로 하니씩 개선하자.
- 이러한 지표가 추가될 때 `호손 효과`로 인해 좋은 효과를 얻을 수 있다.
- 하지만 무언가를 측정할 수 있다고 해서 그것이 무조건 의미 있는 통찰을 제공한다는 뜻은 아니다. 지표를 프로젝트의 목표와 연결하고 지표의 추세를 보면서 시간이 지남에 따라 나아지고 있는지, 나빠지고 있는지 보면서 조정을하자.

<br>

## 간결함의 가치

- 간결함이란 단순히 짧은 코드를 의미하는게 아니라, 필요한 것만 남기고 군더더기를 없애는 것이다.
- 같은 동작을 하더라도 표현이 명확하고 압축적이어야 한다.
- 불필요한 분기, 중복된 로직, 과도한 추상화는 코드의 의도를 흐린다.

```java
// 복잡함
if (user != null) {
    if (user.getAge() != null) {
        if (user.getAge() >= 18) {
            return true;
        }
    }
}
return false;

// 간결함 (표현을 압축)
return user != null 
    && user.getAge() != null 
    && user.getAge() >= 18;
```

<br>

## 단순함의 가치

- `KISS 원칙`을 생각해보면, `단순함`이란 코드를 읽기 쉽고 유지보수하기 쉽게 만드는 것이다. 나는 여기서의 단순함을 `명확성`이라고 생각한다.
- 다만 단순함을 잘못 받아들이면 오히려 더 큰 문제를 만들 수 있다. 아키텍처 관점에서의 사례를 들어보자.

```java
// 복잡함
if (user != null) {
    if (user.getAge() != null) {
        if (user.getAge() >= 18) {
            return true;
        }
    }
}
return false;

// 단순함 (분기를 풀어냄)
if (user == null) return false;
if (user.getAge() == null) return false;
return user.getAge() >= 18;
```

<br>

#### 사례: 상태 불일치를 보정 배치로 해결하려 할 때

- A(예: 레디스)와 B(예: DB)의 상태 불일치 문제를 해결하기 위해 N분 주기의 보정 배치를 도입한다고 가정해보자.
- B를 단일 진실 공급원(Single Source of Truth)으로 두고, 보정 배치가 A의 상태를 B에 맞춰 변경한다. 얼핏 단순해 보이지만, 보정 배치가 수행되기 전에 인입된 이벤트가 불일치 상태에서 처리되면 과대 지급 또는 과소 지급 문제가 발생할 수 있다.

#### 진짜 단순함이란

- 누군가에게는 단순해 보였던 이 아키텍처가 실제로는 더 큰 복잡성을 숨기고 있었던 셈이다.
- 진짜 단순함은 `이 행위를 어떻게 원자적으로 풀 것인가`, `어디까지를 원자적 단위로 볼 것인가`를 고민하는 데서 나온다. 표면적인 단순함이 아니라, 문제의 본질을 명확히 정의한 위에서 나오는 단순함이 진짜 단순함이라고 생각한다.

<br>

## 응집도와 결합도

### 응집도

- 응집도는 본질적으로 비슷한 것들이 모여있는 것을 의미한다.
- 응집이라는 개념은 여러 개념으로 나눌 수 있는데, 가장 중요한것은 `순차적 응집력`과 `기능적 응집력`이라 생각한다.

<br>

우연적 응집력

- 연관성 없는 것들이 우연히 하나의 모듈로 만들어진 경우

```java
class Utils {
    void sendEmail() { ... }
    int calculateTax() { ... }
    String formatDate() { ... }
}
```

<br>

논리적 응집력

- 논리적으로는 비슷하지만 실제로는 서로 다른 역할을 하는 경우

```java
class InputHandler {
    void handle(int type) {
        if (type == 1) readFromFile();
        else if (type == 2) readFromNetwork();
        else if (type == 3) readFromKeyboard();
    }
}
```

<br>

시간적 응집력

- 같은 시점에 실행되어야 한다는 이유로 묶인 경우

```java
class Startup {
    void initialize() {
        loadConfig();
        connectDatabase();
        warmUpCache();
        registerShutdownHook();
    }
}
```

<br>

절차적 응집력

- 특정 순서대로 실행되어야 한다는 이유로 묶인 경우

```java
class OrderProcess {
    void process() {
        validateInput();    // 순서 1
        checkInventory();   // 순서 2
        logActivity();      // 순서 3 - 다른 목적
    }
}
```

<br>

통신적 응집력

- 같은 데이터에 대해 작업하는 요소들이 묶이 경우

```java
class UserReport {
    void printUserData(User user) {
        printName(user);
        printEmail(user);
        printAddress(user);
    }
}
```

<br>

순차적 응집력

- 한 요소의 출력이 다음 요소의 입력이 되는 경우

```java
class DataPipeline {
    Report process(RawData raw) {
        CleanData cleaned = clean(raw);       // 출력: cleaned
        Analysis analyzed = analyze(cleaned); // cleaned를 입력으로
        Report report = generate(analyzed);   // analyzed를 입력으로
        return report;
    }
}
```

<br>

기능적 응집력

- 모든 요소가 단 하나의 명확한 목적을 달성하기 위해 협력 하는 경우

```java
class CircleCalculator {
    double calculateArea(double radius) {
        return Math.PI * radius * radius;
    }
}
```

<br>

### 결합도

- 결합도는 서로 연관되어 있지 않는 모듈들이 얼마나 의존하고 있는가를 말한다.
- 결합도가 높다면 한 부분을 변경했을 때 관련이 없는 다른 무엇가에 영향을 미치기 때문에 수정이 어려워진다.

<br>

내용 결합도

- 한 모듈이 다른 모듈의 내부 데이터를 직접 참조하거나 수정하는 경우

```java
class A {
    public int internalValue = 10;
}

class B {
    void doSomething(A a) {
        a.internalValue = 999;  // A의 내부를 직접 건드림
    }
}
```

<br>

공통 결합도

- 여러 모듈이 전역 변수나 공유 자원에 의존하는 경우

```java
class GlobalConfig {
    public static String dbUrl;
    public static int timeout;
}

class UserService {
    void save() {
        connect(GlobalConfig.dbUrl);  // 전역 상태에 의존
    }
}

class OrderService {
    void process() {
        wait(GlobalConfig.timeout);   // 같은 전역에 의존
    }
}
```

<br>

외부 결합도

- 모듈이 외부에 지정된 형식이나 프로토콜에 의존하고 있는 경우
- 서비스 레이어에서 이러한 경우가 가장 빈번히 발생하는데, 어러한 경우에는 외부 API를 연동하는 부분을 인프라 레이어로 빼놓고, 중간에 인터페이스를 두어 문제를 해결할 수 있다. 결국 DIP를 적용하자.

```java
class PaymentService {

    void pay(String xmlRequest) {  // 외부 시스템의 XML 형식에 강하게 묶임
        // 외부 API의 XML 스펙이 바뀌면 우리도 바뀌어야 함
    }
}
```

<br>

스탬프 결합도

- 필요한 데이터가 아니라 자료 구조 전체를 넘기는 경우

```java
class TaxCalculator {
    int calculate(User user) {  // User 전체를 넘김
        return user.income * 0.1;
    }
}
```

<br>

데이터 결합도

- 필요한 데이터만 매개변수로 넘기는 경우

```java
class TaxCalculator {
    int calculate(int income) {  // 필요한 값만 받음
        return income * 0.1;
    }
}
```

<br>

## 짧은 메서드 지향

- 메서드 이름에 접속사(and, or)가 포함되어 있다면 해당 메서드가 너무 많은 일을 하고 있다는 냄새를 풍기고 있는것이다.
- 보통 JPA를 사용하다보면 무의식적으로 `findByIdAndXXX` 이런식의 메서드를 많이 사용하는데, 이보다는 그 메서드가 달성하려는 목적이 담겨야 한다.

<br>

## 코드 주석의 문제

- 코드의 주석은 보통 최신화가 잘 이루어지지 않는다. 차라리 주석을 작성하는 것보다 코드를 더 읽기 쉽게 만드는데 시간을 쓰는편이 낫다. 맞다!
- 하지만 주석을 절대로 작성하지 말아야 한다는 것은 아니다.
- 주석은 `무엇을` 했는지가 아니라 `왜` 했는지에 초점을 맞춰야 한다. A대신 왜 B를 선택하게 되었는지 또는 특별히 복잡한 로직을 설명하는 것은 도움이 될 수 있다.

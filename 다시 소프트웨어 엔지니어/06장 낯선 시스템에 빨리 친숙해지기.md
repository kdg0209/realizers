# 낯선 시스템에 빨리 친숙해지기

<br>

## 낯선 코드 이해

#### 프로젝트 이해

- 프로젝트가 누구를 위해 어떤 가치를 제공하는가?
- 이해관계자가 누구인가?

#### 기존 문서 검토

- 프로젝트에 대해 최신 문서가 있다면 실펴보자.
- 하지만 많은 스타트업에서의 문서는 최신화가 되어있지 않거나, 존재하지 않을 가능성이 크다. 이 또한 테스트 코드 또는 API 문서화가 있다면 다행이지만 그렇지 않은 환경에서는 코드가 가장 최신 문서이기 때문에 코드를 살펴보는 경우가 가장 많다.

<br>

### 점진적인 정신 모델 구축

#### 정신 모델

- 정신 모델이란 시스템이 작동하는 방식에 대한 우리 머릿속에 표현된 그림이며, 정신 모델을 만들기 위해서는 시간과 의도적인 연습이 필요하다.

#### 정신 모델 만들기

- 시스템이 동작하는 방식을 하나의 큰 그림으로 만들지 않고, `경계`를 나누어 복잡한 시스템을 다루기 좋은 작은 조각으로 나눠야 한다.
- 결국 요구되는 것은 큰 문제를 어떻게 작은 문제로 나누고 이들을 조합할 수 있는 역량이 필요하다.

#### 어떻게 큰 문제를 작은 문제로 나눌 수 있을까?

- 예를들어 `광고 리워드 지급`이라는 하나의 큰 기능을 이해한다고 해보자. 그럼 전체 시스템을 한번에 파악하는 대신 경계를 나눠 그림을 그릴 수 있다.
  - 요청 유입: 사용자의 리워드 참여 요청은 어떠한 경로로 유입되는가?
  - 달성 조건: 리워드 달성 조건은 무엇이고, 어디서 어떻게 검증되는가?
  - 달성 이후: 조건을 달성하면 무슨 일이 일어나는가?
  - 데이터 변화: 참여자와 광고주의 데이터는 각각 어떻게 변경되는가?

<br>

### 되돌릴 수 있는 작은 변경

- 쉽게 되돌릴 수 있는 변경을 점진적으로 쌓아가는 것은, 시스템에 큰 충격을 주지 않으면서 발전시키는 가장 안정적인 방법이다. 변경이 작을수록 문제가 생겼을 때 원인을 좁히기 쉽고, 되돌리는 비용도 낮다.
- 이 원칙은 커밋 단위에도 그대로 적용할 수 있다. 받은 업무의 범위가 크더라도 이를 어떻게 작은 커밋으로 나눌지 미리 고민하면, 각 커밋이 독립적으로 되돌릴 수 있는 단위가 된다. 이렇게 잘게 나눈 커밋은 리뷰어가 변경 의도를 파악하기 쉽게 하고, 커밋 단위 리뷰 요청이나 AI 기반 코드 분석 등 다양한 방식으로 활용할 수 있다.

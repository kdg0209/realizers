# 자동 테스트: 좋은 소프트웨어를 만드는 최고의 습관

<br>

## 태스트 코드의 이점

### 1. 문서 역할을 한다

- 테스트 코드는 Swagger나 Spring Rest Docs처럼 API 사용자에게 제공되는 문서와는 다른 종류의 문서다.
- 테스트 코드를 통해 만들어진 문서는 **엔지니어를 위한 기능 명세서이자 정책 명세서**의 역할을 한다.
- 예를 들어 `BankAccount`의 활성 상태 정책은 아래 테스트만 봐도 한눈에 들어온다.
    - 노출 상태가 아닌 계좌는 활성 상태가 아니다
    - 만료된 계좌는 활성 상태가 아니다
    - 공유 계좌는 활성 상태가 아니다
    - 만료되지 않았고, 공유되지 않았으며, 노출 상태인 계좌만 활성 상태이다

```java
class BankAccountTest {

    @Test
    void 노출_상태가_아닌_계좌는_활성_상태가_아니다() {
        // ...
    }

    @Test
    void 만료된_계좌는_활성_상태가_아니다() {
        // ...
    }

    @Test
    void 공유_계좌는_활성_상태가_아니다() {
        // Given: 공유 상태인 계좌
        BankAccount account = new BankAccount(
            1L, 
            true,   // isDisplay
            false,  // isExpired
            true    // isShared
        );

        // When: 계좌가 활성 상태인지 확인한다
        boolean isActive = account.isActive();

        // Then: 활성 상태가 아니어야 한다
        assertThat(isActive).isFalse();
    }

    @Test
    void 만료되지_않았고_공유되지_않았으며_노출_상태인_계좌는_활성_상태이다() {
        // Given: 만료되지 않고, 공유되지 않았으며, 노출 상태인 계좌
        BankAccount account = new BankAccount(
            1L, 
            true,   // isDisplay
            false,  // isExpired
            false   // isShared
        );

        // When: 계좌가 활성 상태인지 확인한다
        boolean isActive = account.isActive();

        // Then: 활성 상태여야 한다
        assertThat(isActive).isTrue();
    }
}
```

### 2. 유지보수성 향상

- 테스트 가능성을 고려하면서 코드를 작성하다 보면 자연스럽게 결합도가 무엇인지 생각하게 된다.
- 예를 들어 하나의 클래스가 많은 `import`를 선언하고 있다면, "이 클래스가 너무 많은 책임을 가지고 있는 것은 아닐까?" 하는 의문이 생긴다.
- 이어서 "어떤 역할과 책임을 기준으로 더 나눌 수 있을까?", "나눈 책임들을 위임을 통해 협력시켜 목적을 달성할 수 있을까?"와 같은 사고로 자연스럽게 확장된다.
- 이러한 사고 과정은 결국 `단일 책임 원칙(SRP)`으로 이어지고, 더 나은 코드 설계를 가능하게 만든다.

<br>

### 3. 자신감 상승

- 소프트웨어 개발은 끊임없이 코드를 작성하고, 수정하는 반복적인 과정이다. 이 과정에서 테스트가 없다면 창의적이고 도전적인 해결책을 시도할 수 없다.

<br>

## 자동 테스트 유형

### 단위 테스트

- 나는 단위 테스트에 대해서 `고전파` 입장을 선호한다.
- 단위 테스트는 애플리케이션을 구성하는 하나의 메서드가 올바르게 동작하는지 검증하는 테스트다. 여기서 말하는 메서드는 주로 도메인 레이어에 위치한 메서드를 일컫는다.
- 단위 테스트가 실패하면 보통 특정 메서드나 특정 코드 라인을 가리키기 때문에 디버깅이 수월하다.
- 정리하자면, 단위 테스트는 외부 의존성 없이 하나의 단위(주로 메서드 또는 클래스)가 올바르게 동작하는지 검증하는 테스트다. 대표적으로 도메인 레이어의 비즈니스 로직이 단위 테스트의 주요 대상이 된다.

<br>

### 통합 테스트

- 통합 테스트는 시스템의 각 구성 요소들이 서로 맞물려 올바르게 동작하는지 검증하는 테스트다.
- 일반적으로 서비스 레이어를 대상으로 하며, `@SpringBootTest`로 실제 빈을 띄워 검증하거나, `Mockito`를 활용해 행위 기반 검증을 수행한다.
- 다만 `Mockito`로 협력 객체를 모킹해 테스트하는 경우, 학파에 따라 관점이 다르다. 
  - `런던파`는 이를 단위 테스트로 본다, 
  - `고전파`는 외부 의존성(DB, 외부 API)만 모킹하고, 순수 도메인 객체는 실제 객체를 사용한다. 이 또한 외부 의존성이 없다면 단위 테스트로 본다.

```java
// 도메인
@Getter
public class User {
    
    private final Long id;
    private final int point;

    public User(Long id, int point) {
        this.id = id;
        this.point = point;
    }

    public User usePoint(int amount) {
        if (point < amount) {
            throw new IllegalArgumentException("포인트가 부족합니다");
        }
        return new User(id, point - amount);
    }
}

// 외부 의존성 (DB 접근)
public interface UserRepository {
    User findById(Long id);
    User save(User user);
}

// 협력 객체 (외부 의존성 아님, 순수 도메인 서비스)
public class PointNotifier {
    public String notify(User user, int usedAmount) {
        return user.getId() + " 유저가 " + usedAmount + " 포인트를 사용했습니다";
    }
}

// 테스트 대상
public class UserPointService {
    
    private final UserRepository userRepository;
    private final PointNotifier pointNotifier;

    public String usePoint(Long userId, int amount) {
        User user = userRepository.findById(userId);
        User updatedUser = user.usePoint(amount);
        userRepository.save(updatedUser);
        return pointNotifier.notify(updatedUser, amount);
    }
}
```

#### 🏛 고전파 스타일

```java
class UserPointServiceClassicalTest {

    // UserRepository는 외부 의존성이므로 가짜 구현체 사용
    private final FakeUserRepository userRepository = new FakeUserRepository();

    // PointNotifier는 순수 객체이므로 실제 객체 사용
    private final PointNotifier pointNotifier = new PointNotifier();

    private final UserPointService sut = new UserPointService(userRepository, pointNotifier);

    @Test
    void 포인트를_사용하면_차감된_결과가_저장되고_알림_메시지가_반환된다() {
        // Given: 1000 포인트를 가진 유저
        userRepository.save(new User(1L, 1000));

        // When: 300 포인트 사용
        String message = sut.usePoint(1L, 300);

        // Then: 결과(상태) 검증
        User savedUser = userRepository.findById(1L);
        assertThat(savedUser.getPoint()).isEqualTo(700);
        assertThat(message).isEqualTo("1 유저가 300 포인트를 사용했습니다");
    }
}

// 외부 의존성을 흉내 내는 가짜 구현체
class FakeUserRepository implements UserRepository {
    private final Map<Long, User> store = new HashMap<>();

    @Override
    public User findById(Long id) {
        return store.get(id);
    }

    @Override
    public User save(User user) {
        store.put(user.getId(), user);
        return user;
    }
}
```

#### 🇬🇧 런던파 스타일

```java
class UserPointServiceLondonTest {

    private final UserRepository userRepository = mock(UserRepository.class);
    private final PointNotifier pointNotifier = mock(PointNotifier.class);

    private final UserPointService sut = new UserPointService(userRepository, pointNotifier);

    @Test
    void 포인트를_사용하면_차감된_유저가_저장되고_알림이_호출된다() {
        // Given: 1000 포인트를 가진 유저
        User user = new User(1L, 1000);
        User updatedUser = new User(1L, 700);

        when(userRepository.findById(1L)).thenReturn(user);
        when(userRepository.save(any(User.class))).thenReturn(updatedUser);
        when(pointNotifier.notify(any(User.class), eq(300))).thenReturn("알림 메시지");

        // When
        sut.usePoint(1L, 300);

        // Then: 행위(상호작용) 검증
        verify(userRepository).findById(1L);
        verify(userRepository).save(any(User.class));
        verify(pointNotifier).notify(any(User.class), eq(300));
    }
}
```

#### 📌 고전파와 런던파의 차이점

| 구분 | 고전파 | 런던파 |
|------|--------|--------|
| 모킹 대상 | 외부 의존성만 (`UserRepository`) | 모든 협력 객체 |
| 검증 방식 | 상태 검증 (`getPoint() == 700`) | 행위 검증 (`verify`) |
| 관심사 | 결과가 맞는지 | 호출이 일어났는지 |
| 깨지기 쉬움 | 비교적 안 깨짐 | 내부 구현 바뀌면 깨짐 |
| 리팩토링 내성 | 강함 | 약함 |

### 종단 간 테스트

- 종단 간 테스트(E2E)는 컨트롤러부터 가장 뒷단의 시스템까지 애플리케이션 전체 흐름을 다루는 가장 포괄적인 테스트다.
- E2E 테스트가 실패하면 원인을 파악하고 수정하는 데 상당한 시간이 소요된다.
- 그럼에도 불구하고 E2E 테스트는 시스템 전반의 기능이 올바르게 동작한다는 높은 확신을 준다.

> 개인적으로는 다양한 케이스를 통합 테스트로 촘촘히 구성하는 것이 비용 대비 효과 측면에서 더 효율적이라고 생각한다.

## 테스트하지 말아야 할 것

#### 언어 기능이나 프레임워크 코드

- 이미 충분히 검증된 언어 기능이나 프레임워크 코드는 테스트할 필요가 없다.
- 예를 들어 JPA 자체에 대한 테스트는 불필요하다고 생각한다. 이미 수많은
  실사용을 통해 검증된 라이브러리이기 때문이다.
- 그 시간에 차라리 서비스 레이어의 비즈니스 로직을 더 꼼꼼히 테스트하자.

#### private 메서드

- private 메서드를 직접 테스트하는 것은 피해야 한다.
- 만약 테스트하고 싶은 욕구가 든다면, 그 메서드가 정말 private이 맞는지
  다시 생각해볼 필요가 있다. 별도의 클래스로 분리되어야 할 책임일 수도 있다.

#### 비결정적 테스트

- 외부 서비스(API, 시간, 랜덤 값 등)에 의존해 테스트 결과가 매번
  달라지는 비결정적 테스트는 피해야 한다.
- 같은 입력에 대해 항상 같은 결과가 나와야 신뢰할 수 있는 테스트가 된다.

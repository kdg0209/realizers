## 1. 자바 스레드의 간략한 역사

### 1-1. 자바는 스레드로 만들어진다.

- 자바에서 스레드는 가장 작은 실행 단위
- 스레드는 동일한 주소 공간(`Heap 영역`)을 공유함
- 각 스레드는 Program Counter(PC Register)와, Call Stack을 독립적으로 가지고 있으며, Stack Frame 내부에는 지역 변수등이 있음
- 자바의 스레드 모델은 스레드 스케줄링 및 실행과 관련하여 OS에 의존하고 있고, OS는 높은 실행 효율을 보장하기 위해 스레드에 CPU를 사용할 수 있는 시간을 부여하고, 스레드 사이의 상태 전환도 관리함

#### 예외와 스레드

- 자바에서 모든 스레드는 각자 고유한 `Call Stack`을 가지고 있음
- `Call Stack`에는 스래드가 생성되어 소멸되기 전까지의 모든 메서드 호출 내역이 기록됨

#### 📌 breakpoint 과정

1. IDE의 특정 소스 라인에 breakpoint를 설정
2. JVM은 해당 라인에 대응되는 메서드와 bytecode index 정보를 찾아 등록함
3. Thread 실행됨
4. 현재 실행중인 Thread의 PC의 bytecode index가 breakpoint에 도달
5. JVM이 breakpoint event를 발생시킴
6. 설정된 suspend 정책에 따라 해당 thread 또는 전체 VM이 suspend됨
7. 디버거(IDE)가 event를 전달받아 중단된 위치를 보여줌

#### 💡 LineNumberTable

- **LineNumberTable**은 클래스 전체가 아니라 각 메서드별로 독립적으로 존재함
- 그래서 여러 메서드에서 bytecode index가 0으로 시작하더라도 충돌하지 않음
- 스레드의 `Call Stack`은 아래 정보를 가지고 있음

```txt
Frame
 ├ class
 ├ method
 ├ bytecode index (bci)
 └ local variables
```

#### 💡 Stack Trace 생성 과정

- 예외가 발생하면 JVM은 현재 frame의 method와 bci를 확인함
- 그다음 해당 method의 LineNumberTable을 조회함
- 현재 **bci**가 어느 구간에 속하는지 계산해 소스 라인을 찾음
- 예를 들어 updateDatabase()에서 bci=9이면, LineNumberTable의 line 30: 0 line 31: 10을 기준으로 0 <= bci < 10 구간에 해당하므로 line 30으로 매핑된다.

```txt
현재 frame
class: Main
method: updateDatabase
bytecode index: 9

# updateDatabase의 정보 
LineNumberTable:
      line 30: 0
      line 31: 10
      line 32: 11

```

## 2. 자바 1.0 스레드의 기원

- 자바는 1.0부터 스레드 기능을 내장하여 1996년에 출시되어 현재까지 하위 호환성을 유지하면서 추가적인 기능을 제공하고 있음
- Thread 클래스를 상속받는 방식은 Task와 Thread의 역할이 강하게 결합됨 따라서 새로운 Task를 실행할 때마다 Thread 객체를 생성해줘야함
- Runnable 인터페이스를 사용하면 Task와 Thread의 역할을 분리할 수 있음 따라서 동일한 Runnable 인스턴스를 여러 Thread에서 재사용 가능

```java
public class Main {

    public static void main(String[] args) {

        // Thread 클래스 상속
        Thread t1 = new ThreadExtension("Worker-1");
        t1.start();

        // Runnable 인터페이스 구현
        Thread t2 = new Thread(new RunnableImplementation(), "Worker-2");
        t2.start();

        // 익명 내부 클래스
        Thread t3 = new Thread(new Runnable() {
            @Override
            public void run() {
                System.out.println("Anonymous Thread: " + Thread.currentThread().getName());
            }
        },"Worker-3");
        t3.start();

        // 람다식
        Thread t4 = new Thread(() -> {
            System.out.println("Lambda Thread: " + Thread.currentThread().getName());
        }, "Worker-4");
        t4.start();

        // ExecutorService
        try (ExecutorService executor = Executors.newFixedThreadPool(3)) {
            for (int i = 0; i < 3; i++) {
                executor.submit(() -> System.out.println("Executing Task Thread: " + Thread.currentThread().getName()));
            }
        }
    }

    static class ThreadExtension extends Thread {

        public ThreadExtension(String name) {
            super(name);
        }

        @Override
        public void run() {
            System.out.println("Extended Thread: " + getName());
        }
    }

    static class RunnableImplementation implements Runnable {

        @Override
        public void run() {
            System.out.println("Runnable Thread: " + Thread.currentThread().getName());
        }
    }
}
//
Runnable Thread: Worker-2
Lambda Thread: Worker-4
Extended Thread: Worker-1
Anonymous Thread: Worker-3
Executing Task Thread: pool-1-thread-2
Executing Task Thread: pool-1-thread-1
Executing Task Thread: pool-1-thread-3
```

<br>

#### 💡 Thread의 실행

- Thread의 **start()** 메서드는 스레드를 실행시키는 메서드이며, `system call`을 통해 커널에 스레드 생성을 요청함
- 자바 Thread는 OS 스케줄러에 의해 실행순서가 결정되며, 스레드 실행 시점을 JVM이 제어할 수 없음

<br>

#### 🚗 Thread 실행 과정

1. Main Thread가 새로은 Thread 생성
2. Main Thread가 start 메서드를 통해 스레드 실행시킴
3. start 메서드는 내부적으로 native 메서드은 **start0()** 메서드를 호출하여 커널에게 커널 스레드를 생성해달라고 `system call`을 보냄
4. 커널은 `system call`을 받아 커널 스레드를 생성하고, 커널 스레드는 자바 스레드의 1:1 매핑됨
5. 커널 스레드는 **OS Scheduler**로 부터 CPU를 할당받기 전까지 **실행대기** 상태로 머물게됨
6. 커널 스레드가 **OS Scheduler**에 의해 CPU를 할당받아 실행 상태가 되면 JVM에 매핑된 자바 스레드의 **run** 메서드를 호출하여 task를 실행하게됨

<br>

#### 🧨 주의점

- Thread의 **run** 메서드를 직접 호출한다면 호출한 스레드의 stack위에서 실행되는 것임

<br>

## 3. 스레드에 숨어 있는 비용

- 서블릿 컨테이너를 사용하면 요청당 스레드 모델(`thread-per-request`)을 사용함
- 요청당 하나의 스레드가 서버의 처리를 거친 후 응답에 이르기까지 전체 생애주기를 관리함
- 스레드 개수가 증가되면 context-switching 비용도 증가됨

#### 스레드의 추가 메모리 공간

- 스레드는 `Heap` 영영과 별도로 각 스레드마다 독립적인 **Stack** 영역을 가지며, **Stack** 메모리는 `-Xss` 설정에 따라 달리지지만 보통 수백 KB에서 수 MB 정도의 크기를 사용함. 기본 설정에서는 약 1MB ~ 2MB 수준의 **Stack** 메모리가 할당됨
- 스레드 하나만 생각하면 별문제 없어 보이지만 수천 개의 동시 요청이 발생하는 대규모 애플리케이션에서는 메모리 사용량이 증가될 수 있음

#### 생성 가능한 스레드 개수

- 애플리케이션에서 생성할 수 있는 최대 스레드 개수는 OS의 네이티브 스레드 최대 개수에 의해 제한됨, 즉 OS는 생성할 수 있는 스레드의 최대 한도가 정해져 있음

```java
public class Main {

    public static void main(String[] args) {

        AtomicInteger atomicInteger = new AtomicInteger(0);
        try {
            while (true) {
                Thread thread = new Thread(() -> {
                    atomicInteger.incrementAndGet();
                    LockSupport.park();
                });
                thread.start();
            }
        } catch (OutOfMemoryError e) {
            System.out.println("Thread Limit: " + atomicInteger);
            e.printStackTrace();
        }
    }
}
```

## 4. Lock을 지원하는 여러가지 방법

- `java.util.concurrent.atomic` 패키지 내부에는 동시성 환경에서 값을 안전하게 변경할 수 있는 다양한 클래스를 지원함
- Java에서 동시성을 제어하기 위해 `synchronized`, `ReentrantLock`, `volatile`, `AtomicXXX`을 지원해주고 있는데 이에 대해 살펴보자

<br>

### 4-1. synchronized

- `synchronized` 키워드를 사용하면 객체의 `Mark Word`가 변경되며 Lock 상태가 관리됨
- 객체의 `Mark Word`에는 Lock 상태를 나타내는 `비트(lock bits)` 가 존재함

#### 💡 Lock 상태 변화

1. Unlocked 상태
	- 객체의 `Mark Word`에 기본 메타데이터가 저장됨
 	- 스레드가 Lock을 획득하지 않은 상태

2. Lightweight Lock (Thin Lock)
	- 스레드가 `synchronized` 블록에 진입하면 **CAS**연산을 통해 Mark Word를 스레드의 Lock Record와 연결
 	- 경쟁이 없는 경우 빠르게 Lock 획득 가능

3. Heavyweight Lock (Fat Lock / Monitor)
	- 여러 스레드가 동시에 Lock을 획득하려고하면 Lock이 Monitor 기반의 OS mutex로 확장됨
 	- 이 경우 스레드가 OS 수준에서 block/wakeup 됨

4. Lock 해제
	- 임계 영역을 벗어나면 Lock이 해제되며 `Mark Word`의 Lock 상태가 다시 변경됨

<br>

#### 💡 Mark word란?

- 자바 객체의 `Mark Word`는 객체 헤더(`Object Header`)의 일부로 JVM이 객체의 런타임 메타데이터를 저장하는 공간
- JVM이 32bit에서 실행되면 32bit를 가지고, 64bit에서 실행되면 64bit를 가짐

<br>

#### 💡 JDK 15 이전의 synchronized

- Biased Lock(편향 락)이 사용되었음
- `synchronized` 키워드를 사용하면 모니터를 할당받고 해제하는 과정이 성능상 비효율적일 수 있음
- 이때 `thread id`를 체크하여 **Lock**을 획득하고자 하는 스레드가 동일한 스레드라고 판단되면 OS Lock을 사용하지 않도록 함

```txt
Unlocked
   ↓
Biased Lock (JDK 15 이전)
   ↓
Lightweight Lock
   ↓
Heavyweight Lock (Monitor)
```

<br>

#### 💡 JDK 15 이후의 synchronized

- [Deprecate and Disable Biased Locking](https://openjdk.org/jeps/374)를 보면 15 이후에 Deprecated됨
- 이유는 개선과 최적화가 진행되면서 `Biased Lock`의 이점이 사라짐
	
```txt
Unlocked
   ↓
Lightweight Lock
   ↓
Heavyweight Lock
```

<br>

#### 💡 모니터란?

- `Monitor`는 **Lock**보다 높은 추상화로 임계 영역에 한 번에 하나의 스레드만 진입할 수 있도록 프로그래밍 언어에서 제공하는 동기화 기법을 의미합
- `synchronized` 키워드를 사용할 때 `Monitor`가 활성화되어 동기화가 이루어짐
- 즉, `Monitor`는 특정 객체에 의해 동적으로 사용되며 **Mark Word**의 Lock 정보를 활용하여 동작하게 됨

<br>

### 4-2. ReentrantLock

- `ReentrantLock`은 내부적으로 AQS(AbstractQueuedSynchronizer) 기반으로 구현됨
- Lock 상태는 AQS의 state 필드로 관리됨 (state = 0 → Lock 없음, state > 0 → Lock 획득 상태)

#### 🚗 Lock 획득 과정

```txt
-- 👍 성공 케이스
Thread A lock()
     ↓
CAS(state, 0 → 1) 시도
     ↓
    성공
     ↓
owner = Thread A
     ↓
   락 획득

-- 👎 실패 케이스
Thread B lock()
     ↓
CAS(state, 0 → 1) 시도
     ↓
    실패
     ↓
CLH Queue enqueue (AbstractQueuedSynchronizer의 enqueue)
     ↓
LockSupport.park() # 스레드를 WAITING 상태로 변경
     ↓
   대기 상태

-- 🔓 Lock 해제 과정
Thread A unlock()
     ↓
   state--
     ↓
   state == 0
     ↓
LockSupport.unpark(nextThread) # WAITING 상태의 스레드를 RUNNABLE 상태로 변경
```

<br>

#### 공정성

- `synchronized` 키워드를 사용하면 공정성을 잃을 수도 있음
- 그 이유는 **BLOCKED** 상태의 스레드들 중에서 어떤 스레드가 Lock을 획득할지 모르고 최악의 경우 특정 스레드는 오랜기간 Lock을 획득하지 못 할수도 있음
- `ReentrantLock`은 공정하게 Lock을 획득할 수 있는 기능 제공함

비공정 모드

- 비공정 모드는 `ReentrantLock`의 기본 모드
- 해당 모드에서는 Lock을 먼저 요청한 스레드가 먼저 획득한다는 보장 없음, 즉 대기중인 스레드 중 아무나 Lock 획득 가능

공정 모드

- 공정 모드는 Lock을 요청한 순서대로 스레드가 Lock을 획득함
- 다만 비공정 모드에 비해 성능이 저하됨
- tryLock() 메서드는 비공정 모드이고, tryLock(long time, TimeUnit unit) 메서드는 공정 모드임

<br>

### 4-3. volatile

- 멀티 스레드환경에서 한 스레드가 변경한 값이 다른 스레드에서 언제 보이는지에 대한 문제를 `가시성`이라함

#### 🚗 volatile의 원리

- 공유 변수에 `volatile`를 선언함으로써 CPU가 변수를 읽을 때 캐시 메모리를 거치지 않고 메인 메모리에서 직접 변수를 읽고, 수정된 변수를 즉시 메인 메모리에 반영함

#### 📌 volatile의 한계

- `volatile` 키워드는 가시성은 보장해주지만 **동시성**은 보장해주지 않음
- `volatile` 키워드는 읽기 작업을 하는 스레드가 N개이고, 쓰기 작업을 하는 스레드가 1개라면 동시성을 보장할 수 있지만 N:M의 관계라면 동시성을 보장할 수 없음

### 4-4. AtomicXXX

- `CAS(Compare-And-Swap)`를 이용해 Lock 없이 원자적 연산을 제공

#### 🚗 동작 원리

```txt
value = 5(공유 변수)

# 두 스레드가 동시에 공유 변수를 읽음
Thread A read → 5
Thread B read → 5

# CAS
CAS(memory, expected, newValue)

Thread A가 위 연산을 수행함
CAS(value, 5, 6)

memory(5) == expected(5)
→ 조건 만족
→ value = 6 으로 변경

Thread B가 위 연산을 수행함
CAS(value, 5, 6)

현재 메모리 상태 value = 6
memory(6) == expected(5)
→ 조건 불만족
→ CAS 실패
```

<br>

#### 📌 ABA 문제

- 중간에 값이 바뀌었다가 다시 돌아오는 경우를 감지하지 못함

```txt
초기 상태
value = A

Thread A는 공유 변수를 읽어 C로 변경할 준비함
read → A
expected = A
CAS(value, A, C) 준비

Thread B는 공유 변수를 읽어 값을 B로 변경함
read → A
expected = A
CAS(value, A, B) 성공
value = B

Thread B가 다시 한번 공유 변수를 읽어 값을 A로 변경함
read → B
expected = B
CAS(value, B, A) 성공
value = A

Thread A가 이제 값을 A로 변경함
CAS(value, A, C) 수행

마지막 구간에서 Thread A는 공유 변수를 아무도 건드리지 않았다고 생각하게됨
그러나 사실 A → B → A로 값이 변경된 적이 있음
```

<br>

#### 💡 왜 CAS는 Lock 없이도 thread-safe한가?

- CPU가 `CAS` 연산을 하나의 **atomic instruction**으로 실행하고, **cache coherence protocol**이 동시에 수정되는 것을 방지
- 캐시 일관성 프로토콜(`cache coherence protocol`)은 각 CPU 코어가 메모리의 상태를 일관성 있게 볼 수 있도록 보장함. 하나의 CPU 코어가 캐시 라인에 데이터를 저장하면 **캐시 일관성 프로토콜**에 의해 다른 CPU의 캐시 라인도 무효화되거나 업데이트됨

<br>

## 5. Work-Stealing 알고리즘

- `Work-Stealing`은 **ForkJoinPool**의 핵심 동작 원리로 효율적인 작업 분배를 가능하게함

#### 🚗 작동 원리

1. **ForkJoinPool**은 외부에서 제출된 **ForkJoinTask**를 `Submission Queue`에 넣음
2. 각 **Worker Thread**들이 작업을 가져와 자신의 **Work Queue**에 넣음
3. 스레드는 자신의 **Work Queue**에서 작업을 꺼내 실행하며, 이 과정에서 락이 필요하지 않음
4. 자신의 **Work Queue**가 비게되면 다른 스레드의 **Work Queue**에서 작업을 훔쳐와 실행함, 이때 다른 스레드의 **Work Queue**에 `CAS` 기반 동기화가 이루어짐


<details>
	<summary>JDK 21에서의 ForkJoinPool의 scan 메서드</summary>
	
```java
private int scan(WorkQueue w, int prevSrc, int r) {
	WorkQueue[] qs = queues;
	int n = (w == null || qs == null) ? 0 : qs.length;
	for (int step = (r >>> 16) | 1, i = n; i > 0; --i, r += step) { // 랜덤 순회 시작
		int j, cap; WorkQueue q; ForkJoinTask<?>[] a;
		if ((q = qs[j = r & (n - 1)]) != null &&
			(a = q.array) != null && (cap = a.length) > 0) {
			int src = j | SRC, b = q.base;							 
			int k = (cap - 1) & b, nb = b + 1, nk = (cap - 1) & nb; // steal할 위치 계산
			ForkJoinTask<?> t = a[k];
			U.loadFence();                  // for re-reads
			if (q.base != b)                // inconsistent
				return prevSrc;
			else if (t != null && WorkQueue.casSlotToNull(a, k, t)) { // CAS로 실제 stealing 시도
				q.base = nb;
				w.source = src;
				if (src + (src << SWIDTH) != prevSrc &&
					q.base == nb && a[nk] != null)
					signalWork();           // propagate at most twice/run
				w.topLevelExec(t, q);
				return src + (prevSrc << SWIDTH);
			}
			else if (q.array != a || a[k] != null || a[nk] != null)
				return prevSrc;             // revisit
		}
	}
	return -1;
}
```
</details>


<br>

#### 💡 Work-Stealing의 특징

- 스레드가 자신의 **Work Queue**에서는 작업을 꺼낼 때 `LIFO(Last-In-First-Out)` 방식으로 작업을 처리하지만, 다른 스레드의 **Work Queue**에서 작업을 훔쳐올 때 `FIFO(First-In-First-Out)` 방식으로 훔쳐옴
- `참조 지역성`을 활용한 설계임
	- 최근에 생성된 작업은 메모리상 가까운 위치에 존재할 가능성이 높기 때문에 `LIFO`로 처리하면 CPU 캐시 효율이 높아짐
 	- 반대로 **Stealing** 작업에서는 오래된 작업을 `FIFO`로 처리하면 큰 작업 단위를 분산시켜 스레드간 작업 균형을 유지함

<br>

<details>
	<summary>예제 코드</summary>
	
```java
public class Main {

    public static void main(String[] args) {
        int[] numbers = new int[16];
        for (int i = 0; i < numbers.length; i++) {
            numbers[i] = i + 1;
        }

        ForkJoinPool forkJoinPool = new ForkJoinPool(4);
        SumTask task = new SumTask(numbers, 0, numbers.length);
        int result = forkJoinPool.invoke(task);

        System.out.println();
        System.out.println("=== Result ===");
        System.out.println("sum = " + result);

        System.out.println();
        System.out.println("=== Pool Status ===");
        System.out.println("parallelism = " + forkJoinPool.getParallelism());
        System.out.println("pool size = " + forkJoinPool.getPoolSize());
        System.out.println("active thread count = " + forkJoinPool.getActiveThreadCount());
        System.out.println("steal count = " + forkJoinPool.getStealCount());

        forkJoinPool.shutdown();
    }

    public static class SumTask extends RecursiveTask<Integer> {

        private static final int THRESHOLD = 4;

        private final int[] numbers;
        private final int start;
        private final int end;

        public SumTask(int[] numbers, int start, int end) {
            this.numbers = numbers;
            this.start = start;
            this.end = end;
        }

        @Override
        protected Integer compute() {
            int length = end - start;

            if (length <= THRESHOLD) {
                int sum = 0;
                for (int i = start; i < end; i++) {
                    sum += numbers[i];
                }
                System.out.printf("[%s] task 실행: range=(%d, %d), sum=%d%n", Thread.currentThread().getName(), start, end, sum);
                return sum;
            }

            int mid = (start + end) / 2;
            SumTask leftTask = new SumTask(numbers, start, mid);
            SumTask rightTask = new SumTask(numbers, mid, end);

            System.out.printf("[%s] 작업 분할: (%d, %d) -> (%d, %d) + (%d, %d)%n", Thread.currentThread().getName(), start, end, start, mid, mid, end);

            leftTask.fork();
            int rightResult = rightTask.compute();
            int leftResult = leftTask.join();
            int total = leftResult+ rightResult;

            System.out.printf("[%s] 결과 병합: (%d, %d) -> %d + %d = %d%n", Thread.currentThread().getName(), start, end, leftResult, rightResult, total);

            return total;
        }
    }
}
```

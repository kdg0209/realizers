# 통계 테이블 설계

<br>

## 1. 설계 방법

### 1-1. 미리 계산하여 저장

- 가장 간단한 방법은 통계를 위한 별도의 테이블을 만들어 미리 계산하여 저장하는 방식
- 배치를 사용하여 원본 테이블에서 데이터를 읽어와서 통계 테이블에 적재하는 방식

#### 장점

1. 조회 속도

    > 미리 계산이 되어 있으므로 조회 속도가 빠름

#### 단점

1. 실시간성 부족

    > 배치성이기 때문에 실시간 통계는 확인하기 어려움

2. 관리 포인트 증가

    > 배치 프로그램을 만들어야하니 관리 포인트가 증가함

<br>

### 1-2. 하이브리드 설계

#### 전략

- **과거 데이터**: 이미 집계된 통계 테이블에서 조회
- **오늘 데이터**: 원본 테이블에서 조회
- **결합**: 두 데이터를 `UNION ALL`을 사용하여 합침

#### 역할 분담

- **통계 테이블**: 이미 집계된 데이터이기 때문에 부하가 거의 없음
- **원본 테이블**: 날짜 기반의 인덱스가 있다면 범위 스캔이기 때문에 부하가 적음

<br>

## 2. 멱등성 설계

#### 멱등성이란?

- 연산을 한 번 수행하든, 백 번 수행하든 그 결과가 항상 똑같은 것을 의미함
- 단, 원본 데이터가 변하지 않았어야함

#### 멱등성 설계가 필요한 이유

- 배치가 N번 수행되면 데이터가 뻥튀기가 되는 문제가 발생하는데, 이러한 문제는 `멱등성`을 고려하지 않았기 때문에 발생함

<br>

### 2-1. DELETE-INSERT를 사용한 멱등성 보장

- 기존 데이터를 삭제하고 원본 데이터를 기준으로 다시 수행하는 방식

#### 단점

- 기존 데이터를 `DELETE`하면 DB는 복구를 위해 **Undo Log**를 생성함
- 다시 `INSERT`하면 **Redo Log**를 기록하고 인덱스를 다시 구성함
- 위 과정에서 불필요한 자원 낭비가 발생함
- 잦은 삭제와 삽입은 데이터 페이지의 파편화 유발 가능

<br>

### 2-2. UPSERT를 사용한 멱등성 보장

- 없으면 INSERT, 있으면 UPDATE 방식이나, 배치에서는 `UPSERT` 방식은 주의가 필요함
- 만약 UPDATE가 누적하여 UPDATE를 수행한다면 `멱등성` 깨짐

```sql
-- 멱등성 보장
point = point

-- 멱등성 깨짐
point = point + point
```

<br>

#### ON DUPLICATE KEY UPDATE

- 이 방식은 데이터베이스에 먼저 삽입을 시도하고, 만약 **PK** 또는 **Unique Key**가 이미 테이블에 존재해서 충돌이 발생한다면 `UPDATE`를 수행하는 방식

```sql
INSERT INTO intraday_sales_stats (today_date, total_order_count, total_sales_amount, updated_at)
SELECT * FROM (
  -- 1. 먼저 오늘치 데이터를 집계하는 서브쿼리를 만든다.
  SELECT
    DATE(order_date) as today_date,
    COUNT(*) as total_order_count,
    SUM(total_amount) as total_sales_amount,
    NOW() as updated_at
  FROM orders -- 원본 테이블
  WHERE order_date >= '2026-01-02'
   AND order_status = 'COMPLETED'
  GROUP BY DATE(order_date)
) AS new_stats -- 2. 이 집계 결과에 대해 'new_stats'라는 별칭을 붙인다.
ON DUPLICATE KEY UPDATE
  total_order_count = new_stats.total_order_count,   -- 서브쿼리의 값으로 덮어쓰기
  total_sales_amount = new_stats.total_sales_amount, -- 서브쿼리의 값으로 덮어쓰기
  updated_at = new_stats.updated_at;                 -- 서브쿼리의 시간으로 갱신
```

#### 장점

1. 원자성 보장

    > DELETE와 INSERT 사이에 다른 트랜잭션이 침투할 수 없음

2. 멱등성 보장

    > N번 수행하더라도 항상 같은 결과를 보장

3. 성능

    > DELETE INSERT의 자원 낭비 최소화

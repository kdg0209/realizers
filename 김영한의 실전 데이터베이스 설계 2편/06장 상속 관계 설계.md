# 상속 관계 설계

### 문제 상황

- 공통 속성이 있고, `도서`, `전자제품`, `의류`가 공통 속성을 물려받고, 각자 고유한 속성을 추가로 가지는 상황

#### 공통 속성

- 상품 아이디
- 상품명
- 가격
- 수량
- 등록일

#### 도서만의 속성

- 저자
- 출판사
- ISBN

#### 전자제품만의 속성

- 제조사
- 보증기간
- 전력소비량

#### 의류만의 속성
  
- 사이즈
- 색상
- 소재

<br>

## 1. 구현 클래스마다 테이블 전략: Roll-down 

- 자식 클래스마다 구현 테이블을 만들고, 각 구현에 부모 클래스의 속성을 모두 포함하는 방식

#### 예시

```sql
-- 도서 테이블 (부모 속성 + 도서 고유 속성)
CREATE TABLE book (
  product_id BIGINT PRIMARY KEY AUTO_INCREMENT,
  name VARCHAR(200) NOT NULL,
  price INT NOT NULL,
  stock_quantity INT NOT NULL DEFAULT 0,
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  -- 도서 고유 속성
  author VARCHAR(100) NOT NULL,
  isbn VARCHAR(13),
  publisher VARCHAR(100)
);

-- 전자제품 테이블 (부모 속성 + 전자제품 고유 속성)
CREATE TABLE electronics (
  product_id BIGINT PRIMARY KEY AUTO_INCREMENT,
  name VARCHAR(200) NOT NULL,
  price INT NOT NULL,
  stock_quantity INT NOT NULL DEFAULT 0,
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  -- 전자제품 고유 속성
  manufacturer VARCHAR(100) NOT NULL,
  warranty_months INT NOT NULL DEFAULT 12,
  power_consumption INT
);

-- 의류 테이블 (부모 속성 + 의류 고유 속성)
CREATE TABLE clothing (
  product_id BIGINT PRIMARY KEY AUTO_INCREMENT,
  name VARCHAR(200) NOT NULL,
  price INT NOT NULL,
  stock_quantity INT NOT NULL DEFAULT 0,
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  -- 의류 고유 속성
  size VARCHAR(10) NOT NULL,
  color VARCHAR(50) NOT NULL,
  material VARCHAR(100)
);
```

#### 장점

1. 단순하고 직관적임
2. 각 유형별 조회 성능이 좋음
3. NOT NULL 제약 조건 설정 가능

#### 단점

1. 전체 상품 조회가 어려움
2. 외래키 설정 불가능(`orders` 테이블에서 **product_id**컬럼을 기준으로 FK를 설정하려고 할 때 어떤 테이블을 선택 해야할지 모름)

<br>

## 2. 단일 테이블 전략: Single Table Strategy

- 단일 테이블 전략은 부모와 모든 자식의 모든 속성을 하나의 테이블로 관리하는 방식

#### 예시

```sql
CREATE TABLE product (
  product_id BIGINT PRIMARY KEY AUTO_INCREMENT,
  product_type VARCHAR(20) NOT NULL, -- 'BOOK', 'ELECTRONICS', 'CLOTHING'
 
  -- 공통 속성
  name VARCHAR(200) NOT NULL,
  price INT NOT NULL,
  stock_quantity INT NOT NULL DEFAULT 0,
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
 
  -- 도서 전용 속성
  author VARCHAR(100),
  isbn VARCHAR(13),
  publisher VARCHAR(100),
 
  -- 전자제품 전용 속성
  manufacturer VARCHAR(100),
  warranty_months INT,
  power_consumption INT,
 
  -- 의류 전용 속성
  size VARCHAR(10),
  color VARCHAR(50),
  material VARCHAR(100)
);
```

#### 장점

1. 모든 데이터가 하나의 테이블에 있으므로 부모 테이블과의 조인이 필요 없음
2. 전체 상품 조회시 `UNION ALL`과 같은 설정이 필요 없음
3. 외래 키 설정이 가능함(`orders` 테이블에서 `product` 테이블의 product_id로 FK 설정 가능)
4. 상품 아이디가 유일함
5. 공통 속성 변경이 쉬움

#### 단점

1. NULL 값이 많아짐
2. NOT NULL 제약 조건을 설정할 수 없음
3. 모든 컬럼이 하나의 테이블에 있으므로 테이블이 거대해짐

#### 해당 전략을 선택하면 좋은 경우

1. 상품 유형의 종류가 적고, 앞으로도 크게 늘어나지 않을거라고 기대될 때
2. 각 유형별 고유 속성이 적을 때
3. 전체 상품 조회가 빈번하고 성능이 중요할 때
4. 구조를 단순하게 유지하고 싶을 때

<br>

## 3. 조인 전략: Joined Strategy

- 조인 전략은 부모 테이블과 자식 테이블을 분리하고, 조인을 통해서 데이터를 조회하는 방식
- 가장 정석적이고 정규화된 형태

#### 예시

```sql
-- 부모 테이블: 공통 속성을 가진다
CREATE TABLE product (
  product_id BIGINT PRIMARY KEY AUTO_INCREMENT,
  product_type VARCHAR(20) NOT NULL, -- 구분자 컬럼
  name VARCHAR(200) NOT NULL,
  price INT NOT NULL,
  stock_quantity INT NOT NULL DEFAULT 0,
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  INDEX idx_product_type (product_type)
);

-- 자식 테이블: 도서
CREATE TABLE book (
  product_id BIGINT PRIMARY KEY, -- AUTO_INCREMENT 설정이 아님
  author VARCHAR(100) NOT NULL,
  isbn VARCHAR(13),
  publisher VARCHAR(100),
  FOREIGN KEY (product_id) REFERENCES product(product_id) -- 부모 테이블의 product_id 참조
);

-- 자식 테이블: 전자제품
CREATE TABLE electronics (
  product_id BIGINT PRIMARY KEY, -- AUTO_INCREMENT 설정이 아님
  manufacturer VARCHAR(100) NOT NULL,
  warranty_months INT NOT NULL DEFAULT 12,
  power_consumption INT,
  FOREIGN KEY (product_id) REFERENCES product(product_id) -- 부모 테이블의 product_id 참조
);

-- 자식 테이블: 의류
CREATE TABLE clothing (
  product_id BIGINT PRIMARY KEY, -- AUTO_INCREMENT 설정이 아님
  size VARCHAR(10) NOT NULL,
  color VARCHAR(50) NOT NULL,
  material VARCHAR(100),
  FOREIGN KEY (product_id) REFERENCES product(product_id) -- 부모 테이블의 product_id 참조
);
```

#### 장점

1. 테이블이 정규화되어 있어 데이터 중복이 없음
2. NULL값이 없음
3. NOT NULL 제약조건을 설정할 수 있음
4. 외래 키 설정이 가능함
5. 새로운 유형이 추가될 때 새로운 자식 테이블만 추가하면 됨

#### 단점

1. JOIN이 필요함
2. INSERT시 부모 저장 후 자식 저장해야 함
3. 전체 상품 조회시 JOIN으로 인해 쿼리가 복잡해질 수 있음

#### 해당 전략을 선택하면 좋은 경우

1. 상품 유형이 많거나 앞으로 계속 늘어날 예정일 때
2. 각 유형별 고유 속성이 많을 때
3. 데이터 무결성이 중요할 때
4. 정규화된 구조를 원할 때

<br>

### 실무 가이드

- 실무에서는 데이터의 중요도와 조회 빈도, 확장성을 고려하여 **단일 테이블 전략**과 **조인 전략** 중 적절한 것을 선택해야 함
- 일반적으로 데이터의 구조가 단순하고, 확장 가능성이 낮다면 **단일 테이블 전략**
- 비지니스 로직이 복잡하고 데이터의 정합성이 중요하다면 **조인 전략**

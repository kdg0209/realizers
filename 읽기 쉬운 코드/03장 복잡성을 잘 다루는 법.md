# 복잡성을 잘 다루는 법

- 복잡성을 잘 다루기 위해서는 '지속가능한' 소프트웨어 개발을 해야합니다.
- 여기서 말하는 지속가능성이란 것은 코드를 소유한 조직 자체를 유지하도록 만드는 부분인데 이렇게 표현을 하면 와닿지 않지만 앞으로의 내용을 보면 많은 공감이 될 것입니다.

<br>

## 목적

### 지속가능성

- 조직은 돈을 벌기 위해, 돈을 절약하기 위해 등 다양한 이유로 소프르웨어를 개발 합니다.
- 소프트웨어는 어떤 방식으로든 조직을 지원해야하며, 새로운 기능을 추가하거나, 기존 기능을 개선하거나 하는 작업들은 소프트웨어의 지원을 종료하는 날까지 이어지게 됩니다.
- 아래 마틴 파울러님이 하신 말씀은 정말 많은 개발자에게 와닿는 말이 아닐까 싶습니다. 짤은 작업 시간으로 인해 코드 퀄리티가 떨어지거나, 깨진 창문으로 인해 코드 퀄리티가 떨어지기 시작한다면 그 속도는 걷잡을 수 없이 빨라지는거 같습니다.

```text
- 마틴 마울러 -
내부 품질이 좋지 안으면 처음에는 작업 진행이 빠르지만, 시간이 지남에 따라 새로운 기능을 추가하기가 점점 어려워집니다.
작은 것을 변경하려 해도 프로그래머는 읽기 어려운 코드의 많은 영역을 이해해야 합니다.
어떤 것을 바꿨을 때 의도하지 않게 잘못된 부분이 없는지 테스트 해야하고, 고쳐야할 결함들이 발생합니다.
```

<br>

### 가치

- 소프트웨어는 목적을 달성하기 위해 존재합니다. 소프트웨어는 가치를 제공해야 합니다.
- 가치를 우선순위에 두는것은 당연합니다. 그러나 코드가 항상 가치를 만들어야 한다는 생각은 반대로 가치 없는 코드를 작성하는 것은 의미가 없는 행위인가?라는 논리적 오류로 이어집니다. 하지만 이는 잘못된 생각이고 모든 코드가 직접적으로 측정 가능한 가치를 만드는 것은 아니기 때문입니다.
이는 아키텍처를 예로들면 제대로된 아키텍처가 없어서 발생하는 문제는 항상 뒤늦게 발견되곤 합니다. 이렇듯 코드의 가치는 장기적인 관점에서 평가되어야 합니다.

## 프로그래밍이 어려운 이유

### 코드는 작성하는 것보다 읽는 경우가 더 많다

- 우리는 개발을 하다보면 이미 작성되어 있는 코드에 수정을 한다던가, 팀원의 코드에 대해 리뷰를 한다던가, 버그 수정을 위해 코드를 읽는다던가, 오픈소스를 파악한다던가 등 코드를 작성하는 것보다 읽는 시간이 더 많습니다. 그렇기 때문에 '우리는 가독성 좋게 코드를 작성해야 합니다.'
- 지속가능성을 높이기 위해서는 꾸준히 코드에 관심을 가지고, 가독성 좋은 코드를 작성하여 나중에 보더라도 유지보수하기 쉽게 코드를 작성해야 합니다.

<br>

### 가독성 

- 가독성이 좋은 코드가 무엇일까? 이애대해 스스로 생각해볼 필요가 있습니다.
- 제가 생각하는 가독성이 좋은 코드란 아래와 같습니다.
  - 책을 읽듯 내용이 자연스럽고, 문단이 잘 나눠져 있고, 띄어쓰기가 필요한 곳엔 띄어쓰기가, 쉼표가 필요한 곳에 쉼표가 있듯이 가독성 좋은 코드가 이와 같다고 생각합니다.

<br>

#### 개인적인 생각

- 복장성을 잘 다루기 위해서는 결국 지속가능성이 있어야하고, 이는 꾸준한 관심과 기술적 성숙도에서 비롯된다고 생각합니다. 또한 팀원들도 그에대한 동의가 있어야하지 않을까 생각합니다.

#### 좋은 글

```text
- 56p
많은 코드를 빠르게 만들어내는 건 읽어야 할 코드가 그만큼 더 많아진다는 뜻입니다.
즉, 코드를 많이 만들어낼수록 읽어야 할 코드의 양이 늘어나고, 책임이 증가됩니다.(책임은 내 개인생각)

- 61p [마틴 파울러]
컴퓨터가 인식 가능한 코드는 바보라도 작성할 수 있지만, 인간이 이해할 수 있는 코드는 실력 있는 프로그래머만 작성할 수 있다.

- 62p [J.B 라인스베르거]
우리는 조금 느려질 필요가 있습니다. 입력 속도가 빠를수록 모든 사람이 유지 관리해야 하는 코드가 늘어납니다. 코드는 자산이 아니라, 책임입니다.
```
